1. EXECUTIVE SUMMARY
================================================================================

SizeBox is a mature Unity-based sandbox game with approximately 171 C# source
files (~30,000+ lines of code), 71 Lua behavior scripts, VR support (Oculus +
SteamVR), deprecated UNet multiplayer networking, Lua modding via MoonSharp,
advanced IK animation via RootMotion FinalIK, MMD character model support,
procedural city generation, and a full AI behavior system.

The handoff contains THREE folders:

  1. A compiled Win64 build (playable, version 3.01)
  2. A content creation SDK (Unity 2017.3.0f3 project for exporting models)
  3. A scripts-only source code dump (C# files + .meta files, NOT a full project)

CRITICAL FINDING: The source code folder is NOT a complete Unity project. It
contains only the C# scripts and their .meta files. The full Unity project --
including scenes (.unity), prefabs (.prefab), shaders (.shader), materials,
textures, audio, ProjectSettings, and all asset references -- is MISSING.

This means you CANNOT simply open the source code folder in Unity and hit Play.
Significant reconstruction work is required.

However, the combination of the built game (which contains compiled assets and
Lua scripts), the SDK (which is a full Unity project and establishes the Unity
version), and the source code (which has the complete C# codebase) provides
enough material to reconstruct and continue development -- with effort.

5. DO I NEED UNITY?
================================================================================

  YES. Absolutely.

  SizeBox is a Unity game. Every aspect of it -- scenes, prefabs, materials,
  shaders, physics, rendering, UI, asset management, build pipeline -- depends
  on the Unity Editor. There is no way to continue development without Unity.

  Specifically, you need Unity to:
    - Reconstruct the project from source code + built assets
    - Edit scenes, prefabs, materials, and other serialized assets
    - Compile and test C# scripts in context
    - Build the game executable
    - Use the SDK's model export pipeline
    - Debug and profile the game

  The Lua behavior scripts can be edited with any text editor (they are plain
  .lua files loaded at runtime), but this only covers AI behaviors -- not core
  game systems, rendering, physics, or UI.

================================================================================

  RECOMMENDED STARTING VERSION: Unity 2017.3.0f3

  WHY:
    1. CONFIRMED: The SDK's ProjectVersion.txt specifies exactly 2017.3.0f3
    2. The source code uses APIs consistent with Unity 2017:
       - UnityEngine.Networking (UNet -- removed in later versions)
       - UnityEngine.VR namespace (renamed to UnityEngine.XR in 2017.2+)
       - Conditional compilation: #if ENABLE_VR
    3. The built game's modular DLL structure (80+ UnityEngine.*.dll) matches
       Unity 2017.3's module organization
    4. Opening the SDK in this exact version is guaranteed to work (Library/ exists)
    5. The source code will compile without API migration issues

  MIGRATION CONSIDERATIONS:
    - Unity 2017.3 is very old (released Dec 2017). It lacks:
      - Modern render pipelines (URP/HDRP)
      - Addressables asset system
      - Modern Input System (though Unity.InputSystem.dll is present in build)
      - Many quality-of-life improvements
    - UNet was deprecated in Unity 2018 and removed later. Upgrading Unity
      versions WILL break the networking code.
    - UnityEngine.VR was renamed to UnityEngine.XR. Migration required for VR
      code in newer Unity versions.
    - Post-processing v1 was replaced by v2/v3 in later Unity versions.

  DOWNLOAD: Unity 2017.3.0f3 is available from Unity's archive:
    Unity Download Archive > Unity 2017.x

  ALTERNATIVE: If you plan to modernize eventually, consider:
    - Phase 1: Get running on 2017.3.0f3 first (safest)
    - Phase 2: Migrate to Unity 2019.4 LTS (last version with UNet support)
    - Phase 3: Replace UNet with Mirror or Netcode for GameObjects
    - Phase 4: Migrate to Unity 2021/2022 LTS

7. ARCHITECTURE OVERVIEW
================================================================================

  The game follows a SINGLETON-HEAVY, COMPONENT-BASED architecture typical of
  Unity 2017-era projects. Key architectural patterns:

  DESIGN PATTERNS USED:
    - Singleton (GameController, IOManager, ObjectManager, StateManager,
      MainController)
    - Strategy (IDecisionMaker -> GTSDecisionMaker, MicroDecisionMaker)
    - Observer (EventManager pub/sub, SyncVar hooks)
    - Command (Action queue execution in AIController)
    - Factory (Behavior creation from Lua via MoonSharp)
    - MVC (UI system: View/, Controller/, Model/ folders)
    - Component (Unity's AddComponent<T> throughout)
    - State Machine (MainController scene states, BehaviorQueue)

  CLASS HIERARCHY:
    MonoBehaviour
    +-- EntityBase (NetworkBehaviour)
        +-- Humanoid
        |   +-- Giantess (giant characters)
        +-- Micro (tiny characters)
            +-- Player (player-controlled character)
            +-- MicroNPC (AI-controlled tiny characters)

  CORE MANAGERS (Singletons):
    - GameController: Global game state, settings, player tracking
    - MainController: Scene management and transitions
    - IOManager: Runtime asset loading (models, audio, shaders)
    - ObjectManager: Network entity spawning (UNet)
    - EventManager: Global pub/sub event system
    - StateManager: Game state persistence
    - SoundManager: Audio management
    - LuaManager: Lua script loading and execution
    - PreferencesCentral: User settings/preferences

  DUAL SCRIPTING ARCHITECTURE:
    The game uses a hybrid C#/Lua architecture:
    - C# handles core systems (physics, rendering, networking, IK, UI)
    - Lua handles AI behaviors (loaded from StreamingAssets at runtime)
    - LuaManager + 18 LuaBinding classes bridge the two layers
    - MoonSharp interpreter provides Lua 5.2 compliance

9. RISKS, MISSING PIECES, AND LIKELY BREAKAGES
================================================================================

  ---- CRITICAL RISKS ----

  [RISK-01] INCOMPLETE PROJECT (SEVERITY: CRITICAL)
    The Sizebox_SourecCode folder is NOT a full Unity project. It is missing:
      - All .unity scene files (9 scenes referenced in build)
      - All .prefab files
      - All .shader files
      - All .material files
      - All textures and audio assets
      - ProjectSettings/ folder
      - Packages/ folder
      - Any asset bundles
    IMPACT: Cannot open in Unity as-is. Full project reconstruction required.
    EVIDENCE: No Assets/, ProjectSettings/, Library/ folders in source dump.

  [RISK-02] DEPRECATED NETWORKING (SEVERITY: HIGH)
    The game uses UNet (UnityEngine.Networking), which was deprecated in Unity
    2018.x and progressively removed. Key classes affected:
      - EntityBase extends NetworkBehaviour
      - SyncVar attributes on network-synced properties
      - ClientScene.RegisterSpawnHandler for entity spawning
      - NetworkManager singleton pattern
    IMPACT: Cannot upgrade past Unity 2019.4 without replacing entire
    networking layer. Migration to Mirror or Netcode for GameObjects required.
    EVIDENCE: "using UnityEngine.Networking;" in 7+ files

  [RISK-03] PAID ASSET STORE DEPENDENCIES (SEVERITY: HIGH)
    RootMotion FinalIK is a PAID Unity Asset Store package ($90+).
      - Used extensively in GtsScripts/IK/ (6 files)
      - Classes: FullBodyBipedIK, LookAtIK, GrounderFBBIK
    Without a valid license or the original package files, the IK system
    will not compile.
    EVIDENCE: "using RootMotion.FinalIK;" in GiantessIK.cs

  [RISK-04] LEGACY VR API (SEVERITY: MEDIUM)
    Uses UnityEngine.VR namespace (renamed to UnityEngine.XR in Unity 2017.2+).
    VR code uses:
      - VRSettings.LoadDeviceByName() (deprecated)
      - VRSettings.enabled (deprecated)
    The #if ENABLE_VR preprocessor guards help, but VR will need migration
    to XR Plugin Management for modern Unity versions.
    EVIDENCE: Camera/VRCamera.cs uses UnityEngine.VR

  [RISK-05] MMD4MECANIM AVAILABILITY (SEVERITY: MEDIUM)
    MMD4Mecanim may no longer be available or maintained for modern Unity.
    It was a specialized plugin for MikuMikuDance model support.
    The DLLs are present in the SDK and build, but source is not available.
    EVIDENCE: MMD4MecanimInternal.dll in both SDK and build

  ---- MODERATE RISKS ----

  [RISK-06] NO VERSION CONTROL HISTORY
    The project has no .git folder. There is no commit history, no branches,
    no record of what changed between versions or why.
    IMPACT: Cannot understand evolution of the code. Cannot revert changes.
    Cannot identify who wrote what or when.

  [RISK-07] SINGLETON-HEAVY ARCHITECTURE
    Nearly every major system uses a singleton pattern. This creates tight
    coupling and makes testing, refactoring, and feature isolation difficult.
    Not a blocker, but increases maintenance burden.

  [RISK-08] UNITY 2017 LIMITATIONS
    Unity 2017.3 lacks many modern features:
      - No Addressables (asset management)
      - No Universal Render Pipeline
      - No modern Input System (though DLL present in build)
      - No Package Manager (uses UnityPackageManager, the predecessor)
      - Limited C# language version (no async/await, no pattern matching)

  [RISK-09] POTENTIAL ASSET MISMATCH
    The built game may have been built from a DIFFERENT version of the source
    code than what's in Sizebox_SourecCode. There's no way to verify they
    match without the original Unity project.

  [RISK-10] MONO SCRIPTING BACKEND SECURITY
    The Mono build means Assembly-CSharp.dll in the built game can be
    decompiled with tools like ILSpy or dnSpy. This is useful for verification
    but means the shipped code is essentially open.

  ---- MISSING PIECES ----

  [MISSING-01] Original Unity project (scenes, prefabs, materials, etc.)
  [MISSING-02] RootMotion FinalIK asset package or license
  [MISSING-03] Git history / version control
  [MISSING-04] Build pipeline configuration / CI/CD

    Step 2: Create a new empty Unity project
    Step 3: Copy ProjectSettings from the SDK project as a starting baseline
    Step 4: Copy all 171 .cs files + .meta files into Assets/Scripts/
    Step 5: Acquire and import required packages:

              - MMD4Mecanim (from SDK project's Assets/MMD4Mecanim/)

              - Cubed's Unity Shaders (from SDK)

            and creating new scenes with the correct object hierarchies
    Step 9: Set up the Models/ runtime loading folder structure
    Step 10: Attempt to compile and resolve errors iteratively

  RISK: Medium-High (scenes and prefabs must be reconstructed manually)

  If the source code is incomplete or doesn't match the build:

    Step 1: Use UABE (Unity Asset Bundle Extractor) or AssetRipper to

            and scripts from the compiled build
    Step 3: Use the extracted project as a starting point
    Step 4: Replace decompiled scripts with the actual source code
    Step 5: Fix up any references that broke during extraction

  ESTIMATED EFFORT: 1-3 weeks
  RISK: Medium (extraction tools may not perfectly reconstruct all references)

  ---- PATH C: LUA-ONLY MODDING (MINIMAL EFFORT) ----

  If you only want to modify AI behaviors without touching C# code:

    Step 1: Edit Lua files directly in the built game's StreamingAssets/lua/
    Step 2: Run the game and test changes (Lua is loaded at runtime)
    Step 3: No Unity installation needed for this path

  ESTIMATED EFFORT: Immediate
  LIMITATION: Only AI behaviors can be modified. No access to core systems.
